# DSI SQL Implementation Guide

## Overview

This guide explains how to use the T-SQL script generated by the DSI data retrieval application to
populate a SQL Server database with comprehensive user, organization, and service data from the DfE
Sign-In platform.

---

## Database Architecture

### Schema: `dsi`

The generated script creates a normalized relational schema with the following tables:

```
dsi.Users (1)
    ├── dsi.UserOrganisationMappings (many)
    │   └── dsi.Organisations (many)
    │       └── dsi.Services (many)
    │           └── dsi.Roles (many)
    │               └── dsi.UserServiceRoles (many)
    └── dsi.UserServiceRoles (many)
```

### Table Definitions

#### `dsi.Users`

Primary user information from DfE Sign-In.

| Column      | Type             | Notes                                                  |
| ----------- | ---------------- | ------------------------------------------------------ |
| UserId (PK) | UNIQUEIDENTIFIER | User's unique identifier in DfE Sign-In                |
| Email       | NVARCHAR(255)    | User's email (unique identifier in DfE Sign-In)        |
| GivenName   | NVARCHAR(100)    | First name                                             |
| FamilyName  | NVARCHAR(100)    | Last name                                              |
| UserStatus  | INT              | User status (1 = active, etc.)                         |
| UserSource  | NVARCHAR(50)     | Origin: 'DfE Sign In' (seed) or 'Discovered' (via API) |
| CreatedAt   | DATETIME         | Record creation timestamp                              |

**Indexes:**

- `IX_Users_Email` on Email
- `IX_Users_Source` on UserSource

**Notes:**

- `UserSource = 'DfE Sign In'` indicates users from the seed CSV file
- `UserSource = 'Discovered'` indicates users found via the paginated `/users` endpoint (only if
  `DISCOVERY_MODE=true`)
- The UserSource field allows auditing and distinguishing between directly referenced and discovered
  users

#### `dsi.UserOrganisationMappings`

Links users to organizations with role information.

| Column                         | Type             | Notes                          |
| ------------------------------ | ---------------- | ------------------------------ |
| UserOrganisationMappingId (PK) | UNIQUEIDENTIFIER | Auto-generated                 |
| UserId (FK)                    | UNIQUEIDENTIFIER | References `dsi.Users`         |
| OrganisationId (FK)            | UNIQUEIDENTIFIER | References `dsi.Organisations` |
| OrgRoleId                      | INT              | Organization role identifier   |
| OrgRoleName                    | NVARCHAR(255)    | Role name (e.g., "Approver")   |
| CreatedAt                      | DATETIME         | Record creation timestamp      |

**Indexes:**

- `IX_UserOrg_UserId` on UserId
- `IX_UserOrg_OrgId` on OrganisationId

**Notes:**

- Users can have multiple organizations
- Each user-org relationship has an associated role

---

#### `dsi.Services`

Services available within organizations.

| Column              | Type             | Notes                          |
| ------------------- | ---------------- | ------------------------------ |
| ServiceId (PK)      | UNIQUEIDENTIFIER | Auto-generated                 |
| OrganisationId (FK) | UNIQUEIDENTIFIER | References `dsi.Organisations` |
| ServiceName         | NVARCHAR(255)    | Service name                   |
| ServiceDescription  | NVARCHAR(MAX)    | Service description            |
| CreatedAt           | DATETIME         | Record creation timestamp      |

**Indexes:** `IX_Services_OrgId` on OrganisationId

**Notes:**

- Example services: "DfE Sign-in", "Plan Technology for Your School"
- Services are organization-specific

---

#### `dsi.Roles`

Roles available within services.

| Column         | Type             | Notes                      |
| -------------- | ---------------- | -------------------------- |
| RoleId (PK)    | UNIQUEIDENTIFIER | Auto-generated             |
| ServiceId (FK) | UNIQUEIDENTIFIER | References `dsi.Services`  |
| RoleName       | NVARCHAR(255)    | Role name                  |
| RoleCode       | NVARCHAR(100)    | Machine-readable role code |
| CreatedAt      | DATETIME         | Record creation timestamp  |

**Indexes:** `IX_Roles_ServiceId` on ServiceId

**Notes:**

- Roles are service-specific
- Example role codes: "role1-code", "role2-code"

---

#### `dsi.UserServiceRoles`

Junction table for user roles in services.

| Column                 | Type             | Notes                     |
| ---------------------- | ---------------- | ------------------------- |
| UserServiceRoleId (PK) | UNIQUEIDENTIFIER | Auto-generated            |
| UserId (FK)            | UNIQUEIDENTIFIER | References `dsi.Users`    |
| ServiceId (FK)         | UNIQUEIDENTIFIER | References `dsi.Services` |
| RoleId (FK)            | UNIQUEIDENTIFIER | References `dsi.Roles`    |
| CreatedAt              | DATETIME         | Record creation timestamp |

**Indexes:**

- `IX_UserServiceRole_UserId` on UserId
- `IX_UserServiceRole_ServiceId` on ServiceId
- `IX_UserServiceRole_RoleId` on RoleId

**Notes:**

- Allows multiple roles per user in a service
- Supports complex role hierarchies

## Using the Generated SQL Script

### Step 1: Prepare the Script

1. Generate the SQL script by running the Node application:

   ```bash
   node index.js
   ```

2. Open the generated `.sql` file (e.g., `outputs-20250605T1630.sql`) in SQL Server Management
   Studio or VS Code with the SQL extension.

### Step 2: Configure the Target Database

Edit the first line of the script:

```sql
USE [YourDatabaseName];
GO
```

Replace `YourDatabaseName` with your target database name. The script will:

- Create the `dsi` schema if it doesn't exist
- Drop and recreate all tables (safe for repeated runs)

### Step 3: Review the Script (Optional)

The script includes:

- **Schema creation** with DROP/CREATE logic
- **User inserts** with comment headers showing email
- **Organization inserts** with all provider data
- **Relationship inserts** for mappings, services, roles
- **Summary query** at the end

### Step 4: Execute the Script

```sql
-- Execute entire script
-- This can take 1-2 minutes depending on user/org volume
```

**Execution time factors:**

- Number of users: ~5-10ms per user
- Number of organizations: ~2-5ms per org
- Number of services/roles: ~1-2ms per item

### Step 5: Verify Results

The script ends with a summary query:

```sql
SELECT 'Users' as EntityType, COUNT(*) as Count FROM dsi.Users
UNION ALL
SELECT 'Organisations', COUNT(*) FROM dsi.Organisations
UNION ALL
SELECT 'Services', COUNT(*) FROM dsi.Services
UNION ALL
SELECT 'Roles', COUNT(*) FROM dsi.Roles
UNION ALL
SELECT 'User-Service Roles', COUNT(*) FROM dsi.UserServiceRoles;
```

This shows you:

- Total users imported
- Total organizations
- Total services
- Total roles
- Total user-service-role mappings

## Common Queries

### Find all organizations for a user

```sql
SELECT
    o.Name,
    o.URN,
    o.UKPRN,
    uom.OrgRoleName
FROM dsi.Users u
JOIN dsi.UserOrganisationMappings uom ON u.UserId = uom.UserId
JOIN dsi.Organisations o ON uom.OrganisationId = o.OrganisationId
WHERE u.Email = 'user@example.com';
```

### Find all users in an organization

```sql
SELECT
    u.Email,
    u.GivenName,
    u.FamilyName,
    uom.OrgRoleName
FROM dsi.Organisations o
JOIN dsi.UserOrganisationMappings uom ON o.OrganisationId = uom.OrganisationId
JOIN dsi.Users u ON uom.UserId = u.UserId
WHERE o.Name = 'Your Organisation Name'
ORDER BY u.Email;
```

### Find all services and roles for a user in an organization

```sql
SELECT
    s.ServiceName,
    r.RoleName,
    r.RoleCode
FROM dsi.Users u
JOIN dsi.UserServiceRoles usr ON u.UserId = usr.UserId
JOIN dsi.Services s ON usr.ServiceId = s.ServiceId
JOIN dsi.Roles r ON usr.RoleId = r.RoleId
JOIN dsi.Organisations o ON s.OrganisationId = o.OrganisationId
WHERE u.Email = 'user@example.com'
  AND o.Name = 'Your Organisation Name';
```

### Find organizations by provider type

```sql
SELECT
    Name,
    UKPRN,
    ProviderTypeName,
    SourceSystem,
    MasterProviderStatusName
FROM dsi.Organisations
WHERE ProviderTypeName LIKE '%Academy%'
  OR SourceSystem = 'GIAS'
ORDER BY Name;
```

### Find all schools (GIAS establishments)

```sql
SELECT
    Name,
    URN,
    UKPRN,
    Address,
    CategoryName,
    StatusName
FROM dsi.Organisations
WHERE SourceSystem = 'GIAS'
  AND CategoryId = '001'  -- Establishment
  AND Status = 1  -- Open
ORDER BY Name;
```

### Find users with "Approver" role

```sql
SELECT
    u.Email,
    u.GivenName,
    u.FamilyName,
    o.Name as OrganisationName,
    s.ServiceName
FROM dsi.Users u
JOIN dsi.UserOrganisationMappings uom ON u.UserId = uom.UserId
JOIN dsi.Organisations o ON uom.OrganisationId = o.OrganisationId
JOIN dsi.Services s ON o.OrganisationId = s.OrganisationId
WHERE uom.OrgRoleName = 'Approver'
ORDER BY u.Email, o.Name;
```

## Re-running the Script

The generated SQL script is **idempotent** - you can run it multiple times safely:

1. **Drop and recreate approach**: The script drops all tables and recreates them with fresh data
2. **Safe for updates**: If you run the data collection again with new/updated user data, simply:
   - Run the Node application again
   - Execute the new SQL script
   - All tables will be refreshed with latest data

## Performance Considerations

### Indexing

The script creates indexes on:

- Foreign key columns (for JOIN performance)
- Common search columns (Email, UKPRN, UPIN)

For large datasets (1000+ users):

- Consider additional indexes on `dsi.Organisations.Name`
- Consider index on `dsi.Services.ServiceName` if frequently queried

### Data Volume Estimates

- 1000 users → ~50-200 organizations, ~500-5000 services
- Database size: typically 50-500 MB depending on address/description length

### Query Optimization

```sql
-- Good: Use specific WHERE clauses
SELECT * FROM dsi.Users WHERE Email = 'user@example.com';

-- Better: Use indexes
SELECT * FROM dsi.Users WHERE UserId = '123e4567-e89b-12d3-a456-426614174000';

-- For large result sets, consider:
SELECT * INTO dsi_users_export FROM dsi.Users;
```

## Backup and Recovery

Before running the script in production:

```sql
-- Backup the dsi schema (recommended)
IF OBJECT_ID('dsi_backup', 'U') IS NOT NULL
  DROP TABLE dsi_backup;

SELECT * INTO dsi_users_backup FROM dsi.Users;
SELECT * INTO dsi_organisations_backup FROM dsi.Organisations;
-- ... repeat for other tables as needed
```

Recovery (if needed):

```sql
TRUNCATE TABLE dsi.Users;
INSERT INTO dsi.Users SELECT * FROM dsi_users_backup;
```

## Troubleshooting

### Issue: Script fails with "Schema creation error"

**Solution:** Ensure user has `db_owner` or equivalent permissions:

```sql
EXEC sp_addrolemember 'db_owner', 'your_login';
```

### Issue: Foreign key constraint errors

**Solution:** This usually indicates data integrity issues. Check for:

```sql
-- Find orphaned records
SELECT * FROM dsi.UserOrganisationMappings
WHERE OrganisationId NOT IN (SELECT OrganisationId FROM dsi.Organisations);
```

### Issue: NULL values in expected columns

**Solution:** This is normal - NULL indicates data not available from API. Check source:

```sql
SELECT
    COUNT(*) as Total,
    COUNT(Address) as HasAddress,
    COUNT(URN) as HasURN
FROM dsi.Organisations;
```

### Issue: Duplicate data after re-running script

**Solution:** This is expected - the script DROPs tables. To preserve historical data:

```sql
-- Rename old tables before running new script
IF OBJECT_ID('dsi.Users', 'U') IS NOT NULL
  EXEC sp_rename 'dsi.Users', 'Users_20250605';
```
